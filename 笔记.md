# 搜索正文提取机制

## 正文来源

正文存在 `node["text"]` 中，有两个来源：

### 1. FTS 正常搜索（主路径）

```
索引阶段：Rust Parser 解析文件 → Node { title, summary, text } → 存入 SQLite
搜索阶段：FTS5 MATCH → 找到 node_id → 从内存中的 Document 树取 node.text.clone()
         → 通过 SearchResult.text 返回给 Python
```

`text` 是索引时**完整存入**的，没有截断。长度取决于 Parser 对文件的分段结果（一个 Markdown 的 `##` 章节就是一个 Node，`text` 就是该章节的完整正文）。

### 2. ripgrep 降级搜索

没有用 FTS 索引，直接读源文件，取匹配行**前后各 5-6 行**的上下文（`ripgrep.py:131-133`）：

```python
context_start = max(0, matched_line - 6)
context_end = min(len(all_lines), matched_line + 5)
text = ''.join(all_lines[context_start:context_end]).rstrip()
```

## 提取后的截断策略

两条路径拿到 `text` 后，截断方式不同：

| | `/s` 终端渲染 | `search_kb` 返回给 LLM |
|---|---|---|
| **截断方式** | 按行提取关键词上下文 | 按字符数截断 |
| **逻辑** | 找含关键词最多的行，取前后各 5 行作为上下文，每行截断到 78 字符 | `_truncate_to_paragraphs(node_text, 800)` 单条最多 800 字符，总计不超 10000 字符 |
| **代码位置** | `cortex_cli.py:295-335` | `kb_tools.py:103, 215` |
| **按什么截断** | 按行，优先展示匹配行附近的行 | 按段落边界，在 800 字符内尽量保留完整段落 |

### `/s` 的行级上下文策略（`cortex_cli.py:298-335`）

```
1. 给每行计算包含几个关键词
2. 选含 >=2 个关键词的行作为"最佳行"
3. 每个最佳行前后各取 5 行（range(-5, 6)）
4. 最多取 3 个最佳行展开上下文
5. 每行渲染时 truncate_ansi_safe(hl_line, 78) 限制 78 字符
```

### `search_kb` 的段落截断策略（`kb_tools.py:173-187`）

```
1. 如果 text <= 800 字符 → 全部保留
2. 截断到 800 字符，然后找最后一个 \n\n（段落边界）
3. 如果段落边界在后半段 → 在段落处截断
4. 否则找最后一个 \n（行边界）
5. 否则硬截断
```

## `/s` 和 `search_kb` 搜索结果差异

两者共享相同的搜索管线（FTS → 邻近度评分 → 综合评分 → 排序），但 **FTS 阶段的 `max_results` 不同**：

- `/s`：`idx_manager.search(query, max_results=None)` → 回退到配置默认值 20
- `search_kb`：`idx_manager.search(query, max_results=10)` → 工具参数默认值 10

FTS 阶段只返回 BM25 分最高的 N 个节点。`search_kb` 只拿到前 10 个，池子更小，可能导致排在 FTS 后段但综合评分更高的结果被遗漏。
